# CSIE MVP Runtime

## Architecture Overview

The CSIE MVP is an additive semantic reasoning layer inside the existing STRATUM runtime. It does not replace routers, frontend behavior, model training, or existing response fields. The MVP validates a pure-Python categorical flow:

Profile -> STRATUM -> CSIE Types -> Category Layer -> Sheaf Layer -> Bootstrap Layer -> Cech Layer -> Diagnostics -> Final Response

The implementation follows the CSIE technical specification at MVP scale. It represents meaning through semantic objects, morphisms, contexts, sections, gluing checks, H0 global consistency, and H1 residual inconsistency diagnostics. It intentionally does not implement AXIOM, KRONOS, causal functors, model compression, model scaling, transformer attention extraction, sparse LSQR, or frontend redesign.

## Module Descriptions

`backend/services/stratum_omnis/csie_types.py`

Defines the shared CSIE data structures used by the MVP, including semantic objects, morphisms, sections, contexts, profiles, and relation type validation.

`backend/services/stratum_omnis/csie_category.py`

Implements the category world model as semantic objects and morphisms. It supports object insertion, morphism insertion, hom lookup, reachability, and morphism composition.

`backend/services/stratum_omnis/csie_sheaf.py`

Implements context, section, and sheaf storage. It supports deterministic context identifiers, deterministic vectors, section insertion, covering expansion, neighbor retrieval, and gluing validation.

`backend/services/stratum_omnis/csie_bootstrap.py`

Converts existing STRATUM profile dictionaries into CSIE category and sheaf structures. It reuses STRATUM profile fields such as identity anchors, metadata source, situational country, risk level, and tech stack.

`backend/services/stratum_omnis/csie_cech.py`

Implements the MVP Cech layer. It computes cover overlaps, H0-style global sections, H1-style residual conflicts, knowledge gaps, polysemy candidates, uncertainty levels, and diagnostic summaries.

`backend/services/stratum_omnis/csie.py`

Integrates CSIE into the STRATUM runtime response. It preserves legacy fields and adds CSIE engine metadata, covering, global section summary, H1 residual, uncertainty level, and diagnostics.

`backend/services/stratum_omnis/model_hub.py`

Exposes CSIE engine status in the existing model hub registry without adding model downloads, training code, or inference engines.

## Runtime Flow

1. STRATUM loads public-safe profiles through the existing profile store.
2. `build_csie_from_profiles()` creates a category and sheaf from those profiles.
3. Each profile conversion creates an identity object, context nodes, morphisms, and semantic sections.
4. The sheaf expands profile contexts into a covering that includes parent contexts.
5. `CechSolver.validate_global_consistency()` checks overlaps, gluing, H0 sections, H1 residual conflicts, gaps, polysemy, and uncertainty.
6. `get_csie_snapshot()` returns the existing STRATUM classification fields plus additive CSIE runtime fields.

## API Response Shape

Each classification preserves the existing fields:

```json
{
  "stratum_id": "",
  "semantic_objects": [],
  "morphisms": [],
  "context": {}
}
```

Each classification also includes additive CSIE fields:

```json
{
  "csie_engine": {
    "mode": "mvp_cech",
    "version": "day4"
  },
  "covering": [],
  "global_section_summary": {
    "count": 0,
    "concept_ids": []
  },
  "h1_residual": 0.0,
  "uncertainty_level": "LOW",
  "diagnostics": {
    "global_section_count": 0,
    "conflict_count": 0,
    "knowledge_gap_count": 0,
    "polysemy_count": 0,
    "uncertainty": "LOW"
  }
}
```

The top-level snapshot also exposes:

```json
{
  "csie_engine": {
    "mode": "mvp_cech",
    "version": "day4",
    "available": true
  }
}
```

## Diagnostics

Diagnostics are generated by the Cech layer and attached to runtime responses. They are intended to make uncertainty auditable rather than hidden.

- `global_section_count`: number of globally consistent sections returned by H0 computation.
- `conflict_count`: number of failed overlap gluings represented as H1 conflicts.
- `knowledge_gap_count`: number of required context or section gaps.
- `polysemy_count`: number of identifiers with divergent meanings across contexts.
- `uncertainty`: final uncertainty level.

## H0 Meaning

H0 represents globally consistent knowledge. In the MVP, a section is considered globally valid when it can be glued across the requested covering without contradictions or required missing sections. H0 sections become the `global_section_summary` in runtime output.

An empty H0 result is allowed and meaningful. It indicates that the current covering does not contain enough consistent local sections to form a global section.

## H1 Meaning

H1 represents unresolved inconsistency. In the MVP, H1 is reported when overlapping contexts contain shared concepts that fail the gluing threshold. The solver returns conflicts with source context, target context, concept id, similarity, threshold, and reason.

The runtime exposes this as `h1_residual`. A value of `0.0` means no residual conflicts were detected. A nonzero value means unresolved inconsistency remains.

## Uncertainty Levels

`LOW`

No conflicts, no gaps, and no H1 residual.

`MEDIUM`

One to three conflicts or one to three gaps, with no nontrivial H1 residual forcing high uncertainty.

`HIGH`

Four or more conflicts, four or more gaps, or any nontrivial H1 residual.

## MVP Cech Semantics vs Future Sparse Solver

The CSIE specification describes a sparse linear Cech solver using LSQR for a future model-integrated implementation. The MVP intentionally uses a dependency-light pure-Python interpretation over existing Day 2 sheaf structures. This keeps the release production-safe, testable, and compatible with the current STRATUM backend.

Future sparse LSQR work belongs after the MVP and should not be mixed into this release gate.

## Troubleshooting

If `global_section_summary.count` is `0`, inspect `diagnostics.knowledge_gap_count` and `diagnostics.conflict_count`. Empty H0 often means the covering lacks required sections or gluing failed.

If `uncertainty_level` is `HIGH`, inspect `h1_residual`, `diagnostics.conflict_count`, and `diagnostics.knowledge_gap_count`. High uncertainty is expected when conflicts or gaps remain unresolved.

If a legacy client breaks, verify that `stratum_id`, `semantic_objects`, `morphisms`, and `context` are still present. CSIE fields are additive and must not replace the original response shape.

If `/health` or `/status` return 404, verify that `backend/main.py` includes root aliases delegating to the existing status router.

If model hub CSIE metadata is missing, verify that `stratum.csie.v1` includes the `engine` block from `get_csie_engine_status()`.
